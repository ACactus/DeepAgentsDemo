# 项目协作规则

## 代码风格

- 优先按后端服务的思路处理问题，保持代码简单、清晰、可维护。
- 偏好明确的类型边界，接口、配置、运行时上下文等关键位置应有清晰类型约束。
- 强类型应服务于真实约束，不为了“看起来更强类型”引入无意义包装、层级或转发结构。
- 避免过度工程化，不做不必要的分层、间接调用、提前抽象和过早优化。
- 有官方 SDK、标准库或成熟第三方库可用时，优先使用，不重复造轮子。
- 命名要准确，职责边界要清晰，方法和类尽量保持单一职责。

## Python 约定

- 遵守 Python 生态习惯；函数级说明使用 docstring，不要写成函数体内的普通 `#` 注释。
- 注释应克制，只解释代码本身难以直接表达的关键约定、真实映射或业务边界。
- 注释要说人话，优先给真实例子、真实落点和真实映射，避免只解释抽象概念。
- 不要用大段注释污染核心配置；说明应放在最贴近被解释逻辑的位置。

## 设计约定

- 区分业务标识、运行时标识和持久化标识，不要把不同层次的概念混在同一个变量或字段里。
- 设计 key、namespace、path、id 等标识时，优先使用简洁、稳定、顺序明确的分段。
- 如果分段顺序本身已经是清晰协议，不要额外加入无业务价值的自描述标签。
- 对外接口应表达业务语义；内部派生字段应尽量收敛在服务端，不让调用方承担实现细节。
- 修改持久化结构、上下文结构或跨模块约定时，要同步更新最贴近实现处的说明。

## 取舍原则

- 在多个可行方案之间，优先选择简单、务实、落地性强的方案。
- 如果一个抽象只是在转发参数或包装已有能力，优先删除它，直接表达真实意图。
- 不鼓励过度防御性编程；如果防御性代码明显增加理解成本，应选择更直接、可控的实现。
